This specification is titled "Dark Mode Toggle Specification" and must
capture ONLY the behavior needed to add a light/dark theme toggle to the
three existing pages (login, signup, dashboard). It is a PURELY ADDITIVE,
PRESENTATIONAL feature: it changes only how the pages look and must NOT
alter authentication, routing, data handling, escaping, or any of the 8
intentional vulnerabilities.

Core design constraints the spec must lock in:

- Theme is expressed by a `data-theme` attribute on the root `<html>`
  element: absent/"light" = light theme, "dark" = dark theme.
- All themed colors derive from CSS custom properties so both themes
  share the one existing styles.css. No second stylesheet, no build step,
  no framework, no new dependencies.
- The chosen theme is persisted client-side only, in
  localStorage under the key "theme" with values "light" or "dark".
  Nothing is stored server-side, in the session, or in a cookie.
- When no saved value exists, fall back to the OS preference via
  `window.matchMedia('(prefers-color-scheme: dark)')`. The fallback must
  NOT write to localStorage — only an explicit user toggle persists.
- Theme must be restored BEFORE first paint via a small inline script in
  `<head>` (before the stylesheet link), to prevent a flash of the wrong
  theme (FOUC).
- The toggle is a native `<button>` in the shared header (left of the
  logos) so it is focusable and operable with Enter/Space, with a visible
  focus indicator in both themes.
- The button's `aria-label` always names the action it will perform next
  ("Switch to dark theme" while light is active, "Switch to light theme"
  while dark is active) and updates on every switch.
